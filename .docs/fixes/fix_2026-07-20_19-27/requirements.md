## Исходный текст

Пользователь попросил поднять dev-сборку и изменить поведение камня:

- при открытии страницы камень должен начинать падение через 400ms, а не по scroll;
- требование про перерасчёт позиций рук относительно центра/ширины камня удалено, позиционирование рук нужно оставить как сейчас;
- камень должен менять размер в зависимости от высоты: выше — больше, ниже — меньше;
- верхний размер должен быть в 3 раза больше текущего базового, нижний — в 3 раза меньше текущего базового;
- кривая изменения размера должна задаваться через cubic-bezier и быть вынесена в UI;
- изначально камень должен располагаться по середине видимой сцены.

## Функциональные требования

- **Начальное центрирование камня:** Система должна при открытии и загрузке страницы изначально располагать камень по центру видимой сцены.
  - **Пояснение:** Пользователь должен видеть камень в центре до старта первого падения.
  - **Источник:** "Изначально располагай камень по середине"
  - **Acceptance hint:** Открыть страницу без scroll: камень появляется по центру видимой области.

- **Автоматический старт первого падения:** Система должна запускать первое падение камня автоматически через 400ms после готовности сцены.
  - **Пояснение:** Первое падение больше не зависит от первого scroll пользователя.
  - **Источник:** "При открытии и загрузке страницы камень падает вниз через 400ms, а не по scroll."
  - **Acceptance hint:** Открыть страницу и не скроллить: примерно через 400ms камень начинает падать.

- **Scroll не запускает первое падение:** Система должна игнорировать scroll как триггер первого падения и не запускать первое падение повторно.
  - **Пояснение:** Scroll может сохранять существующую механику после первого падения, но не должен быть стартовым триггером.
  - **Источник:** "а не по scroll"
  - **Acceptance hint:** Быстро проскроллить страницу сразу после открытия: камень не стартует раньше 400ms и не получает повторный запуск.

- **Масштаб камня от высоты:** Система должна изменять визуальный размер камня в зависимости от вертикального положения: чем выше камень, тем он больше.
  - **Пояснение:** Размер камня должен быть связан с высотой его положения на сцене.
  - **Источник:** "Камень увеличивается и уменьшается в зависимости от высоты: чем выше, тем камень больше."
  - **Acceptance hint:** При движении вверх камень визуально растёт, при движении вниз уменьшается.

- **Диапазон масштаба камня:** Система должна устанавливать максимальный размер камня в верхнем положении в 3 раза больше текущего базового размера, а минимальный размер в нижнем положении — в 3 раза меньше текущего базового размера.
  - **Пояснение:** Текущий базовый размер считается масштабом `1`.
  - **Источник:** "Максимальный размер камня для самого верхнего положения - в 3 раза больше чем сейчас, минимальное - в 3 раза меньше чем сейчас."
  - **Acceptance hint:** Вверху CSS scale близок к `3`, внизу — к `0.333`.

- **Cubic-bezier для масштаба:** Система должна рассчитывать изменение масштаба камня по кривой cubic-bezier.
  - **Пояснение:** Кривая задаёт нелинейный характер увеличения и уменьшения.
  - **Источник:** "Кривую увеличения в зависимости от высоты сделай в виде кривой cubic-bezier"
  - **Acceptance hint:** Изменение значения cubic-bezier меняет характер масштабирования.

- **Настройка кривой в UI:** Система должна показывать настройку cubic-bezier масштаба камня в UI и применять её live.
  - **Пояснение:** Пользователь может подбирать кривую без изменения кода.
  - **Источник:** "и помести на UI"
  - **Acceptance hint:** В панели настроек есть поле для cubic-bezier масштаба камня; изменение значения влияет на поведение без перезагрузки.

- **Позиционирование рук без изменений:** Система должна оставить текущую модель позиционирования рук без перерасчёта относительно центра камня или ширины фотографии камня.
  - **Пояснение:** Пользователь явно убрал исходное требование про изменение модели координат рук.
  - **Источник:** "Убираю требовние ... оставь как есть сейчас"
  - **Acceptance hint:** Код синхронизации и отрисовки рук не меняет модель координат сверх уже существующей.

## Нефункциональные требования

- **Плавное масштабирование:** Система должна масштабировать камень плавно, без заметных рывков и без смещения визуального центра.
  - **Пояснение:** Масштаб зависит от высоты и должен выглядеть как часть движения, а не как отдельный скачок.
  - **Источник:** "Камень увеличивается и уменьшается в зависимости от высоты"
  - **Acceptance hint:** При движении камня масштаб меняется непрерывно, центр камня визуально остаётся стабильным.

- **Адаптивное начальное положение:** Система должна центрировать камень корректно на разных размерах экрана.
  - **Пояснение:** Начальная середина должна считаться от текущей видимой области.
  - **Источник:** "Изначально располагай камень по середине"
  - **Acceptance hint:** На мобильной и desktop-ширине камень появляется по центру видимой сцены.

- **Стабильность первого таймера:** Система должна запускать автоматическое первое падение один раз и стабильно работать в одиночном и collaborative-режиме.
  - **Пояснение:** Таймер не должен создавать дубли стартов или рассинхронизацию.
  - **Источник:** "камень падает вниз через 400ms"
  - **Acceptance hint:** В solo и collaborative-сценариях первое падение начинается один раз.

- **Минимальность изменения механик:** Система должна сохранить существующую физику, drag-логику, scroll-механику после первого падения и позиционирование рук, кроме изменений, необходимых для утверждённых требований.
  - **Пояснение:** Изменения должны быть локальными к старту, начальному положению, масштабу и UI настройки кривой.
  - **Источник:** "оставь как есть сейчас"
  - **Acceptance hint:** Существующие тесты физики и синхронизации продолжают проходить.

## Проблемы и замечания, превращённые в требования

- **Первое падение привязано к scroll:** Сейчас первое падение запускается по scroll. Требуемое поведение: первое падение запускается автоматически через 400ms после готовности сцены.
  - **Источник:** "камень падает вниз через 400ms, а не по scroll"
  - **Acceptance hint:** Без scroll камень начинает падение; scroll до таймера не запускает падение раньше.

- **Камень не центрирован изначально:** Сейчас начальное положение не формализовано как центр видимой сцены. Требуемое поведение: перед первым падением камень стоит по центру видимой сцены.
  - **Источник:** "Изначально располагай камень по середине"
  - **Acceptance hint:** На загрузке центр камня совпадает с центром viewport/видимой сцены.

- **Размер камня фиксирован:** Сейчас камень не меняет размер по утверждённой кривой высоты. Требуемое поведение: scale камня рассчитывается от вертикальной позиции по cubic-bezier в диапазоне от `1/3` до `3`.
  - **Источник:** "Камень увеличивается и уменьшается в зависимости от высоты"
  - **Acceptance hint:** В коде есть единая модель масштаба; визуально камень больше сверху и меньше снизу.
