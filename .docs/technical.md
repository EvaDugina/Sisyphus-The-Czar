## Архитектура

- **Компоненты:** React-клиент, Vite build/dev proxy, HTTP/WebSocket-сервер, менеджер сессий, файловый session store и общее ядро физики.
- **Потоки данных:** при загрузке без `session` клиент автоматически создаёт комнату через HTTP и выполняет полную навигацию через `location.replace` на `/?session=<id>`; новая страница сразу стартует в shared-режиме и подключается по WebSocket. Затем она отправляет управление и каноническое положение указателя не чаще 30 раз в секунду; сервер рассчитывает физику, рассылает снимки 20 раз в секунду и ретранслирует курсоры.
- **Внешние зависимости:** nginx хоста завершает TLS и проксирует HTTP/WebSocket. БД, очередь и внешние API не используются; постоянное состояние хранится в локальном Docker volume.

Сервер считает физику фиксированным шагом 60 Гц в канонической сцене `1000×2000`, а браузер показывает её на странице высотой `200vh`. Браузеры преобразуют координаты в локальную геометрию и рисуют между снимками через 90-миллисекундную интерполяцию; управляющий браузер при отпускании 150 мс сглаживает возврат от локальной руки к авторитетному серверному состоянию. При ручном отпускании и авто-выскальзывании общее ядро умножает вектор скорости руки на `сила / масса`, влияние рывка и `инерция / 100`; общий пропорциональный предел сохраняет направление вектора. Скорость старше 150 мс считается нулевой, поэтому неподвижная рука не создаёт устаревший бросок. Воздушное затухание горизонтальной скорости постоянно, а параметр «Трение земли» на нижней земле прикладывает силу против горизонтальной скорости и гасит инерцию с учётом тяготения.

При первом intro-захвате сервер сохраняет канонический отпечаток `{x,y,toleranceX,toleranceY}`; попадание удерживаемого камня в отпечаток включает светлую тему, локальный слой сильного ливня и ставит таймер авто-выскальзывания на паузу только до отпускания. После `pointerup` камень не фиксируется: тема снова становится тёмной, дождь скрывается CSS fade-out по локальным параметрам браузера, скорость в отпечатке обнуляется, и дальше камень падает вниз под общей физикой.

## Модули и ответственность

- `server` — HTTP, WebSocket Upgrade, Origin/rate-limit, heartbeat и жизненный цикл процесса.
- `SessionManager` — комнаты, участники, блокировка управления, таймеры, TTL и рассылка снимков.
- `SessionStore` — versioned JSON, атомарная запись через временный файл и восстановление комнат при старте.
- `shared/physics` — чистые расчёты падения, броска, отскока и нормализация параметров.
- `src/components` — React-разметка toolbar, панели параметров, сцены, камня, Canvas и ливня.
- `src/hooks` — lifecycle панели, realtime refs, сцены, следа, дождя и общего experience controller.
- `src/lib` и `src/runtime` — чистые модели координат/status/settings/rain и 60-fps runtime на refs без React-render каждого кадра.

## Данные и контракты

### Сущности

- **Сессия:** случайный 128-битный `id`, состояние камня, общая физика с `physicsVersion`, клиенты, `revision`, скользящий TTL, `emptyDeleteAt` и до 1000 точек траектории. Серверное состояние без WebSocket-клиентов сохраняется в volume; после выхода последнего клиента удаляется через настраиваемый grace, если reconnect не отменил очистку. При восстановлении записи первой версии инерция `0–1` один раз переводится в `0–100`; старый ключ `sliding` переносится в `groundFriction`.
- **Отпечаток:** исходные канонические `x/y` первого касания и допустимое отклонение по двум осям; создаётся один раз, входит в каждый snapshot и переживает reconnect вместе с комнатой.
- **Локальный след:** браузер по умолчанию хранит заданное число точек; личная настройка «Без ограничения длины» отключает локальную обрезку. Серверный буфер остаётся ограничен 1000 точками. Canvas имеет размер только текущего viewport × DPR, а мировые точки смещаются на текущий scroll, поэтому backing buffer не растёт вместе со сценой.
- **Локальный ливень:** сила, CSS timing-function появления/исчезновения и длительности хранятся в `localStorage` вместе с личными визуальными настройками. Они применяются через custom properties `--rain-enter-duration`, `--rain-exit-duration`, `--rain-enter-easing`, `--rain-exit-easing`; изменение силы перезапускает только локальный renderer без изменения состояния комнаты.
- **Состояние камня:** `phase`, `x`, `y`, `vx`, `vy`, `dragging`, `controllerId`, `turbTime`.
- **Клиент:** случайный `clientId` из `sessionStorage`, приватный одноразовый `leaveToken`, последнее `seq`, WebSocket, момент отключения, последний указатель `{x,y,mode,visible}` и время последнего вектора скорости руки.

### API

- `POST /api/sessions` → принимает нормализованные `state`, `physics`, `trail`, опциональный `imprint`; возвращает `sessionId`, `expiresAt`.
- `POST /api/sessions/:id/leave` → принимает `clientId` и персональный `leaveToken`; удаляет клиента и запускает grace пустой комнаты. Повторный запрос к уже удалённой комнате идемпотентен.
- `GET /healthz` → возвращает статус процесса и число комнат.
- `WS /realtime?session=<id>&client=<id>` → оболочка `{v,type,seq,payload}`; поддерживает `control.acquire/move/release`, `pointer.update`, `physics.update`, `session.restart`, `ping`. Первый `control.acquire` передаёт адаптивный допуск отпечатка, а snapshots возвращают его всем участникам; `session.restart` очищает отпечаток, след и блокировку управления в текущей комнате; серверные `pointer.update` и `presence.update` дают другим браузерам текущие `grab/grabbing`-курсоры.

HTTP-тело ограничено 16 КиБ, WebSocket-сообщение — 64 КиБ. Сервер проверяет диапазоны, версию протокола, тип и монотонный `seq`.
Масса валидируется в диапазоне `1–100`, тяготение — `1–10`, инерция — `0–100`, трение земли — `0–1`; значения вне границ сервер обрезает.

## Окружение и конфигурация

| Переменная | Назначение | Пример |
|---|---|---|
| `DEBUG` | dev-режим и послабление same-origin для локальной разработки | `false` |
| `ALLOWED_ORIGIN` | разрешённые публичные origin через запятую | `https://example.com` |
| `SESSION_TTL_SECONDS` | TTL после последней активности | `86400` |
| `EMPTY_SESSION_GRACE_SECONDS` | задержка очистки пустой комнаты для reload/reconnect | `10` |
| `SESSION_STORE_PATH` | путь к versioned JSON-store | `/app/data/sessions.json` |
| `SESSION_PERSIST_INTERVAL_MS` | период фоновой записи движущихся комнат, мс | `250` |
| `PORT` | внутренний порт контейнера | `8080` |

- `DEBUG=true` — Vite на `8080` через 150-миллисекундный polling видит изменения bind mounts Docker Desktop, обновляет React/CSS/assets через Fast Refresh/HMR и проксирует API/WebSocket во внутренний Express на `8081`. `shared/physics.js` вызывает полный reload браузера с 600-миллисекундной задержкой на restart backend, а Nodemon в legacy-watch режиме автоматически перезапускает Express при изменениях `server/` и `shared/`; внешний HMR WebSocket использует порт `18082`. Dispose React runtime снимает listeners, отменяет timers/fetch и закрывает старый realtime socket без повторного reconnect.
- `DEBUG=false` — same-origin/allowlist, CSP, CORP, production-кэш ассетов и без автообновления.

## Развёртывание

- **Локальный запуск:** `docker compose -f docker-compose.dev.yml up --build`; после первого запуска контейнер остаётся работающим, а изменения frontend/backend/shared исходников подхватываются из bind mounts без ручного restart или пересборки. Пересборка нужна только после изменения зависимостей, Dockerfile или Compose-конфигурации.
- **Деплой:** `bash deploy.sh` идемпотентно собирает production-образ, обновляет один контейнер и показывает его статус.
- Приложение публикуется только на `127.0.0.1:18082`; HTTPS/WSS и WebSocket Upgrade настраиваются в nginx хоста. Named volume `/app/data` переживает обычный restart/recreate/deploy контейнера.

Требования для двух пользователей: Ubuntu Server 24.04 LTS, 1 vCPU, минимум 512 МБ RAM, 2 ГБ диска, канал от 1 Мбит/с. Рекомендуются 1 ГБ RAM, 10 Мбит/с, RTT до 100 мс, jitter до 30 мс. Запускается один экземпляр: горизонтальное масштабирование без общего хранилища запрещено.

## Тестирование

- **Сценарии:** физика и таймеры; направление/масштаб инерции при ручном отпускании и авто-выскальзывании; трение земли как торможение на нижней земле; миграция старой шкалы и старого ключа `sliding`; конкурентный захват/TTL/reconnect; HTTP+WebSocket двух клиентов; реальный путь двух браузеров; локальные настройки ливня в панели параметров; возврат камня в отпечаток с включением и настраиваемым скрытием ливня; 10-минутная передача управления с контролем RSS.
- **Команды:** `npm run lint`, `npm run build`, `npm test`, `npm run test:smoke` и перед крупным релизом `npm run test:soak`, только внутри Docker-образов из README.
- **Целевое покрытие:** процент не отслеживается; критический совместный путь обязателен.

При dropped frames клиент ограничивает один визуальный шаг передачи от локальной руки к серверному состоянию, поэтому handoff остаётся плавным даже под нагрузкой.

Финальный прогон 2026-07-15: unit/integration `46/46`, browser smoke `4/4`; двухбраузерный soak `1/1` за 10,8 минуты.

## Безопасность

- Секреты и production-конфигурация только в `.env`; `.env` исключён из git и Docker build context.
- Session ID генерируется из 128 криптографически случайных бит.
- Проверяются Origin, размеры сообщений, типы и диапазоны; создание комнат и подключения ограничены по IP.
- Закрытие вкладки отправляет `sendBeacon`; приватный `leaveToken` не позволяет участнику удалить чужое подключение по видимому `clientId`.
- CSP запрещает фреймы, объекты, камеру, микрофон и геолокацию. `Referrer-Policy: no-referrer` не даёт ссылке комнаты уйти к внешнему шрифту.
- Контейнер запускается не от root, read-only, с `no-new-privileges`, лимитами CPU/RAM/PID и loopback-портом.

## Ограничения этапа

- Нет БД, горизонтального масштабирования, репликации store и бэкапов volume.
- Нет аккаунтов, ролей, MFA, аудита действий, продуктовой аналитики и полноценного мониторинга.
- Нет гарантии качества при RTT выше 100 мс или нестабильной мобильной сети.
- Аварийное завершение браузера может не доставить `pagehide` beacon; WebSocket heartbeat обнаруживает обрыв не позднее connection timeout, затем запускается grace пустой комнаты.
- TLS-сертификат и nginx — инфраструктура хоста, не часть репозитория.

## Технические решения

- **2026-07-14:** клиентская физика расходилась бы между экранами → сервер стал авторитетным, координаты переведены в каноническую сцену → требуется постоянный WebSocket.
- **2026-07-14:** отдельные nginx и realtime-контейнеры увеличивали операционную сложность → один Node-процесс отдаёт статику, API и WebSocket → один порт и один deploy.
- **2026-07-14:** потеря комнат при restart контейнера стала неприемлемой → добавлен локальный `SessionStore` с атомарным JSON и Docker volume → тот же ID восстанавливается в одном экземпляре без Redis/PostgreSQL.
- **2026-07-14:** `pagehide` последнего клиента немедленно удалял комнату при reload → добавлен 10-секундный `emptyDeleteAt`, который отменяется новым подключением; активный клиент продлевает TTL.
- **2026-07-14:** обновления UI терялись при закрытом WebSocket → клиент хранит дельту изменённых ключей и отправляет её после reconnect, сохраняя pending-значения при промежуточных snapshots.
- **2026-07-14:** canvas по высоте всей сцены не рисовался стабильно → backing buffer ограничен viewport × DPR, отрисовка переводится на `scrollX/scrollY`.
- **2026-07-15:** монолитный `index.html` мешал безопасно менять интерфейс → разметка перенесена в React-компоненты, lifecycle в hooks, а статические assets собираются Vite без изменения realtime-протокола.
- **2026-07-15:** параметр инерции одновременно означал воздушное затухание и не управлял стартовым броском → шкала изменена на `0–100`, параметр перенесён в единый расчёт отпускания, а старая шкала local/session storage мигрирует при чтении.
- **2026-07-15:** `sliding` называлось скольжением, но фактически отвечало за торможение на дне → поле переименовано в `groundFriction`, старая схема мигрирует, а расчёт стал сухим трением против `vx` на нижней земле.
- **2026-07-15:** Docker Desktop терял файловые события bind mounts, а Fast Refresh оставлял старый reconnect → Vite и Nodemon переведены на polling, runtime получил полный dispose, shared reload синхронизирован с restart Express → исходники применяются без recreate контейнера и без дублирования WebSocket.

## Frontend

- **Стек:** React + JSX, Vite и глобальный CSS, разделённый на base/controls/scene.
- **Безопасность:** секретов и приватных API в браузере нет; запросы идут только на свой origin.
- **Рендер:** React владеет семантической разметкой и редкими UI-состояниями; координаты, интерполяция и Canvas живут в refs/runtime и не вызывают React-render 60 раз в секунду.
- **Состояния:** автоматическое создание, подключение, online, занятое управление, всегда доступное верхнее копирование URL с 400-мс подтверждением, начальные light/scroll-lock, удалённые `grab/grabbing`-курсоры, общий отпечаток, локальный ливень при удержании камня в отпечатке, локальные параметры силы/кривых/длительностей дождя, reconnect без сброса, восстановление после restart, fallback-замена действительно потерянной комнаты и ограниченный/неограниченный локальный след.
- **Адаптивность:** desktop и mobile используют локальную геометрию поверх канонических координат.
